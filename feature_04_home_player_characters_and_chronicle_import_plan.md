# Фича 04: Персонажи игрока на главной и импорт в контексте хроники

## Problem Statement

Сейчас главный экран показывает только список хроник, поэтому игроку неудобно быстро вернуться к своим готовым персонажам:
- персонажи доступны через страницы хроник или прямую ссылку,
- на главной нет персонального списка завершённых листов,
- импорт персонажа доступен с главной и может создать персонажа вне выбранной хроники,
- импорт на странице персонажа перезаписывает текущий лист, хотя пользовательский сценарий фактически ближе к созданию нового персонажа из файла,
- экспорт сейчас сохраняет привязку к хронике через `meta.chronicleId`, из-за чего перенос JSON между хрониками становится неявным и потенциально ошибочным.

В итоге пользовательский flow “экспортировать персонажа -> импортировать в нужную хронику” работает недостаточно предсказуемо.

## Goal

Добавить на главный экран отдельный блок с карточками завершённых персонажей текущего авторизованного пользователя и привести import/export к хроник-ориентированному сценарию:
- на главной показывать только завершённых персонажей пользователя,
- сортировать персонажей по дате создания, сверху самые новые,
- отображать персонажей карточками в стиле страницы хроники,
- оставить текущий блок хроник и создание хроник на главной без добавления кнопки “Создать персонажа” в новый блок,
- убрать импорт с главной и со страницы персонажа,
- добавить импорт JSON на экран хроники,
- при импорте создавать нового персонажа в текущей хронике,
- при экспорте не сохранять привязку персонажа к хронике.

## Non-Goals (для этой фичи)

- Полная ACL-модель доступа к персонажам и хроникам.
- Перенос прямых ссылок на персонажей на owner-only доступ.
- Отображение черновиков персонажей на главной.
- Поиск, фильтры, пагинация и группировка персонажей на главной.
- Импорт в уже существующего персонажа с перезаписью его данных.
- Миграция legacy-персонажей без `createdByUserId` к конкретным пользователям.
- Изменение формата экспорта сверх удаления хроник-специфичных полей.

## Зафиксированные продуктовые решения

- Главный экран сохраняет существующий блок хроник.
- Новый список персонажей находится отдельным блоком.
- В список попадают только персонажи текущего пользователя:
- `createdByUserId === auth.user.id`,
- `creationFinished === true`,
- `deleted !== true`.
- Черновики на главной не показываются.
- Сортировка: `createdAt` по убыванию, то есть сверху самые новые персонажи.
- Карточки визуально опираются на существующий паттерн `character-card` со страницы хроники.
- Клик по карточке открывает `/c/:uuid`.
- Кнопку “Создать персонажа” в блок персонажей на главной не добавляем.
- Импорт доступен только на экране хроники.
- Импорт с главной удаляется.
- Импорт со страницы персонажа удаляется.
- Экспорт остаётся доступен как сейчас на странице персонажа.
- Экспортированный JSON не должен сохранять `meta.chronicleId`.
- При импорте на экране хроники создаётся новый персонаж в этой хронике, затем пользователь переходит на страницу нового персонажа.
- Импортированный персонаж получает текущего пользователя как владельца и игрока, независимо от того, что было в JSON.

## Целевые API-контракты

### Список персонажей пользователя

Рекомендуемый контракт:

- `GET /api/characters?owner=me&creationFinished=true`

Поведение:
- требует авторизацию,
- не поддерживает просмотр чужих owner через query params,
- возвращает компактный список `CharacterSummaryDto[]`,
- фильтрует по `createdByUserId`, `creationFinished` и `deleted`,
- сортирует по `createdAt: -1`.

Поля summary:
- `uuid`,
- `creationFinished`,
- `createdAt`,
- `createdByUserId`,
- `createdByDisplayName`,
- `meta.name`,
- `meta.avatarUrl`,
- `meta.playerName`,
- `meta.clanKey`,
- `meta.sectKey`,
- `meta.generation`,
- `meta.chronicleId`,
- опционально `chronicleName`, если сервер добавит удобное отображаемое поле для карточки.

### Импорт персонажа в хронику

Рекомендуемый REST-прагматичный контракт:

- `POST /api/chronicles/:id/characters/import`

Обоснование:
импорт создаёт нового персонажа внутри конкретной хроники, поэтому endpoint должен быть вложен в ресурс хроники, а не перезаписывать существующего персонажа по `uuid`.

Поведение:
- требует авторизацию,
- проверяет существование хроники,
- принимает JSON экспортированного персонажа,
- игнорирует переносимые identity/ownership/system поля,
- принудительно ставит `meta.chronicleId = :id`,
- принудительно ставит `createdByUserId = auth.user.id`,
- принудительно ставит `createdByDisplayName = auth.user.displayName`,
- принудительно ставит `meta.playerName = auth.user.displayName`,
- валидирует итогового персонажа тем же пайплайном, что текущий import,
- создаёт новую запись `Character`,
- возвращает созданного персонажа или summary, достаточный для перехода на `/c/:uuid`.

Поля, которые импорт должен игнорировать из JSON:
- `uuid`,
- `_id`,
- `__v`,
- `createdByUserId`,
- `createdByDisplayName`,
- `createdAt`,
- `updatedAt`,
- `deleted`,
- `deletedAt`,
- `meta.playerName`,
- `meta.chronicleId`.

### Экспорт персонажа

Текущий контракт сохраняется:

- `GET /api/characters/:uuid/export`

Изменение:
- `sanitizeCharacterForExport()` дополнительно удаляет `meta.chronicleId`.

## Step-by-Step Plan

### Step 1. Аудит текущих точек import/export и карточек персонажей

1. Зафиксировать текущие UI-точки импорта:
- `Home` импортирует JSON и создаёт персонажа без выбранной хроники,
- `CharacterPage` импортирует JSON и перезаписывает текущего персонажа.
2. Зафиксировать текущую точку экспорта:
- `CharacterPage` через `GET /characters/:uuid/export`.
3. Зафиксировать reusable UI-паттерн карточек:
- `ChroniclePage` уже показывает `character-card` с аватаром, именем, кланом, сектой, игроком и поколением.
4. Зафиксировать текущий server transfer helper:
- `apps/server/src/utils/characterTransfer.ts`.

Done criteria:
- Понятно, какие UI-кнопки надо убрать/перенести.
- Понятно, какие поля summary нужны для карточек на главной.
- Понятно, где удалить `meta.chronicleId` из экспортируемого JSON.

### Step 2. Backend: список завершённых персонажей текущего пользователя

1. Добавить `GET /characters` в `apps/server/src/routes/characters.ts` с поддержкой `owner=me` и `creationFinished=true`.
2. В route требовать `req.auth.user`.
3. Фильтровать:
- `createdByUserId: auth.user.id`,
- `creationFinished: true`,
- `deleted: false`.
4. Сортировать:
- `createdAt: -1`.
5. Использовать `presentCharacterList()` для актуального `playerName`.
6. Вернуть компактные поля по аналогии с `/chronicles/:id/characters`.
7. При необходимости добавить `createdAt` и `chronicleName` в клиентский `CharacterSummaryDto`.

Done criteria:
- Главная может получить только завершённых персонажей текущего пользователя.
- Endpoint не раскрывает персонажей других пользователей.
- Результат стабильно отсортирован от новых к старым.

### Step 3. Frontend: блок персонажей на главной

1. В `Home` параллельно загружать:
- хроники через текущий `/chronicles`,
- персонажей через новый `/characters?owner=me&creationFinished=true`.
2. Добавить отдельный state для `characters`.
3. Добавить отдельный loading/error UX без поломки текущего списка хроник.
4. Отрисовать блок “Мои персонажи” отдельной `card`.
5. Использовать карточки в стиле `ChroniclePage`:
- аватар или fallback-буква,
- имя или `(Без имени)`,
- клан,
- секта,
- поколение,
- хроника, если сервер отдаёт `chronicleName`.
6. Клик по карточке ведёт на `/c/:uuid`.
7. Добавить empty state:
- если завершённых персонажей нет, показать короткий текст без CTA на создание.
8. Не добавлять кнопку “Создать персонажа” в этот блок.

Done criteria:
- На главной есть отдельный блок карточек персонажей.
- Показываются только завершённые персонажи пользователя.
- Существующий блок хроник остаётся рабочим.

### Step 4. Backend: экспорт без привязки к хронике

1. Обновить `sanitizeCharacterForExport()`:

- дополнительно удалить `meta.chronicleId`.
2. Проверить, что удаление `meta.chronicleId` не ломает импортную валидацию, потому что новый import endpoint будет подставлять хронику из route params.

Done criteria:
- Экспортированный JSON не содержит `meta.chronicleId`.

### Step 5. Backend: импорт нового персонажа в хронику

4. Санитизировать payload:
- удалить identity/system поля,
- удалить owner/player поля,
- удалить `meta.chronicleId`.
5. Создать нового персонажа с новым `uuid`.
6. Проставить:
- `createdByUserId`,
- `createdByDisplayName`,
- `meta.playerName`,
- `meta.chronicleId`.
7. Прогнать существующую import validation логику:

- `validateAllWizardSteps()` или эквивалентный текущий путь,

8. Сохранить персонажа и вернуть DTO.
9. Старый `POST /characters/:uuid/import` удалить или закрыть для пользовательского flow, чтобы импорт не оставался доступным вне хроники.

Done criteria:
- Импорт из хроники создаёт нового персонажа именно в этой хронике.
- JSON не может перенести персонажа в другую хронику через `meta.chronicleId`.

## Риски и нюансы

- Персонажи без `createdByUserId` не попадут в новый блок на главной; это ожидаемое поведение в рамках фичи.
- Если оставить старый `POST /characters/:uuid/import`, появится скрытый API-путь для перезаписи персонажа; лучше удалить route или явно закрыть его.
- Если импортируемый JSON старого формата содержит `meta.chronicleId`, сервер всё равно должен игнорировать его и использовать только `:id` из URL.
- Если карточка на главной показывает `chronicleName`, нужно аккуратно обработать удалённую или отсутствующую хронику.
