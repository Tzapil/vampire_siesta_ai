# Фича 07: Глобальный каталог неигровых персонажей, привязка к хроникам и NPC в бою

## Problem Statement

Сейчас в проекте есть только два режима работы с участниками игры:

- полноценные персонажи игроков, жёстко привязанные к одной хронике через `Character.meta.chronicleId`,
- ручные противники в бою, которые создаются на месте из имени, Ловкости и Смекалки.

Из-за этого отсутствует важный storyteller-flow:

- нельзя завести переиспользуемого неигрового персонажа один раз и потом применять его в разных хрониках,
- нельзя держать общий каталог NPC с редактируемыми характеристиками,
- на странице хроники нет отдельного списка привязанных NPC,
- нельзя быстро добавлять уже подготовленных NPC в бой,
- повторяющиеся противники в бою не связаны с общим каталогом и каждый раз заводятся вручную,
- текущая модель `Character` не подходит для этой задачи, потому что:
  - привязана к одной хронике,
  - заточена под wizard/game/storyteller flow,
  - использует realtime patching и V20-зависимости, которые для NPC не нужны.

В результате storyteller вынужден либо создавать обычных персонажей под NPC-сценарии, либо каждый раз руками пересобирать противников в бою.

## Goal

Добавить отдельную подсистему неигровых персонажей (NPC), которая даёт:

- глобальный каталог NPC, доступный всем авторизованным пользователям,
- отдельный read-only экран просмотра NPC,
- отдельный экран создания и редактирования NPC в стиле `Режима ведущего`, но без wizard и без realtime,
- many-to-many привязку NPC к хроникам,
- author-only блок NPC на странице хроники,
- добавление привязанных NPC в бой как независимых snapshot-копий,
- удаление конкретной NPC-копии из боя,
- замену текущих ручных противников в бою на NPC-основанный механизм.

## Non-Goals (для этой фичи)

- Полная унификация NPC с моделью `Character`.
- Realtime/sync через Socket.IO для NPC-редактора.
- Мастер создания NPC, freebie-система и wizard-валидация.
- Автоматические V20-пересчёты от клана, поколения и других вампирских зависимостей.
- Импорт/export NPC через JSON.
- Пагинация, фильтры кроме поиска по имени и расширенный каталог-менеджмент.
- Настройка видимости NPC-блока в хронике.
- Общий ACL-рефакторинг всех chronicle/combat действий вне NPC-сценариев.
- Сохранение изменений исходного NPC обратно из боя.

## Зафиксированные продуктовые решения

- На главной странице нет списка NPC.
- На главной есть только CTA `Неигровые персонажи` с иконкой, ведущий на отдельную страницу каталога.
- Каталог NPC доступен всем авторизованным пользователям.
- Любой авторизованный пользователь может:
  - создавать NPC,
  - просматривать NPC,
  - редактировать NPC,
  - удалять NPC.
- Клик по NPC из каталога открывает read-only страницу просмотра.
- На read-only странице есть отдельная кнопка `Редактировать`.
- Создание и редактирование идут на отдельной странице, визуально похожей на `Режим ведущего`.
- NPC не используют autosave:
  - изменения живут локально,
  - запись в БД происходит только по кнопке `Сохранить`.
- После создания нового NPC пользователь попадает на read-only страницу этого NPC.
- При конкурентном редактировании используется `last write wins`.
- NPC являются отдельной сущностью, не `Character`.
- NPC не содержат:
  - `wizard`,
  - `playerName`,
  - встроенную привязку к одной хронике.
- Состав NPC:
  - имя,
  - аватар,
  - опциональные `clan/sect/generation`,
  - `attributes`,
  - `abilities`,
  - опциональные `disciplines`,
  - `virtues`,
  - `health`,
  - `bloodPool`,
  - `willpower`,
  - `humanity`,
  - `notes`.
- Для NPC нет сложной V20-логики:
  - только ручные значения,
  - только простые диапазоны.
- Диапазоны:
  - `attributes`: `1-5`,
  - `abilities`: `0-5`,
  - `disciplines`: `0-5`,
  - `virtues`: `1-5`,
  - `health`: текущий 7-клеточный трек,
  - `willpower`: `0-10`,
  - `humanity`: `0-10`,
  - `bloodPool`: `0-50`.
- Один NPC можно привязать к нескольким хроникам.
- Привязку NPC к хронике можно удалить без удаления самого NPC.
- При удалении NPC:
  - он удаляется глобально,
  - автоматически отвязывается от всех хроник,
  - уже созданные боевые snapshot-копии остаются в боях.
- На странице хроники блок NPC видит только автор хроники.
- Никаких переключателей видимости в рамках этой фичи нет.
- Кнопка `Добавить персонажа в хронику` открывает модалку с поиском по имени.
- В модалке привязки показываются только NPC, ещё не привязанные к текущей хронике.
- На боевом экране ручные противники удаляются из UX и из целевого API-сценария.
- В бой можно добавить только NPC, уже привязанных к этой хронике.
- Выбор NPC для боя происходит через модалку с поиском по имени.
- Добавление NPC в бой создаёт snapshot на момент добавления.
- Дальнейшие изменения боевой копии не меняют исходный NPC.
- Один и тот же NPC можно добавить в бой несколько раз.
- Каждая копия NPC в бою независима:
  - собственное здоровье,
  - собственная инициатива,
  - собственный статус `погиб`.
- Имена копий автоматически нумеруются:
  - `Бандит #1`,
  - `Бандит #2`,
  - и т.д.
- Номер в рамках текущего боя монотонно растёт и не переиспользуется:
  - если удалить `#1` и добавить снова, будет `#3`.
- Удаление NPC из боя удаляет только конкретную боевую копию.
- Если NPC уже добавлен в бой, а затем отвязан от хроники, его боевая копия остаётся до ручного удаления.
- NPC-участников боя видят все пользователи, открывшие бой.
- Но управлять NPC в бою может только автор хроники:
  - добавлять,
  - удалять,
  - менять инициативу,
  - менять здоровье,
  - менять статус `погиб`.
- Порядок списков по умолчанию:
  - глобальный каталог NPC: сначала новые (`createdAt desc`),
  - список NPC в хронике: по имени,
  - модалка привязки к хронике: по имени,
  - модалка добавления в бой: по имени.

## Целевая модель данных

### `npcs`

Новая отдельная коллекция `npcs`.

Рекомендуемая форма:

- `meta`:
  - `name` (required, trim, 1-80),
  - `avatarUrl` (optional, по текущему паттерну `dataUrl`/string),
  - `clanKey` (optional),
  - `sectKey` (optional),
  - `generation` (optional, если задано: `8-14`).
- `traits`:
  - `attributes: Record<string, number>`,
  - `abilities: Record<string, number>`,
  - `disciplines: Record<string, number>`,
  - `virtues: Record<string, number>`.
- `resources`:
  - `bloodPool.current`,
  - `willpower.current`,
  - `humanity.current`,
  - `health: { bashing, lethal, aggravated }`.
- `notes`.
- `createdByUserId`.
- `createdByDisplayName`.
- `deleted`.
- `deletedAt`.
- `createdAt`.
- `updatedAt`.

Ключевое архитектурное решение:

- не переиспользовать `LayeredValue` и не наследовать модель `Character`,
- хранить у NPC простые итоговые числа, потому что для них нет base/freebie/storyteller-слоёв и нет wizard-domain.

### `chronicle_npcs`

Новая таблица-связка many-to-many:

- `chronicleId`,
- `npcId`,
- `createdAt`,
- опционально `addedByUserId`.

Индексы:

- уникальный compound index на `chronicleId + npcId`,
- индекс на `chronicleId`,
- индекс на `npcId`.

Причина отдельной таблицы-связки:

- `Character.meta.chronicleId` годится только для 1:N,
- у NPC нужна нормальная M:N-привязка,
- удаление NPC и отвязка от хроник становятся проще и безопаснее.

### `combat_states`

Текущую поддержку ручных `enemies` нужно заменить на NPC-основанный боевой слой.

Рекомендуемое изменение схемы:

- убрать целевое использование `enemies`,
- добавить `npcs: CombatNpcSnapshot[]`,
- добавить `npcCopyCounters: Record<npcId, number>`.

Форма `CombatNpcSnapshot`:

- `_id`,
- `npcId`,
- `baseName`,
- `displayName`,
- `copyOrdinal`,
- `avatarUrl`,
- опционально `clanKey`,
- опционально `sectKey`,
- опционально `generation`,
- `dexterity`,
- `wits`,
- `health`,
- `dead`,
- `initiative`.

`npcCopyCounters` нужен, чтобы:

- выдавать `#3` после удаления `#1`,
- не переиспользовать номера,
- держать numbering state внутри текущего боя.

## Целевые API-контракты

### Глобальный каталог NPC

- `GET /api/npcs?search=<name>`

Поведение:

- требует авторизацию,
- возвращает список `NpcSummaryDto[]`,
- поиск по имени case-insensitive substring,
- сортировка по `createdAt desc`,
- скрывает `deleted`.

Summary-поля:

- `id`,
- `createdAt`,
- `updatedAt`,
- `meta.name`,
- `meta.avatarUrl`,
- `meta.clanKey`,
- `meta.sectKey`,
- `meta.generation`,
- опционально `createdByDisplayName`.

### Создание NPC

- `POST /api/npcs`

Поведение:

- требует авторизацию,
- принимает полный `NpcInputDto`,
- валидирует диапазоны,
- создаёт NPC,
- возвращает созданный `NpcDto`.

### Просмотр NPC

- `GET /api/npcs/:id`

Поведение:

- требует авторизацию,
- доступен любому авторизованному пользователю,
- возвращает полный `NpcDto`,
- `404`, если NPC не найден или удалён.

### Редактирование NPC

- `PUT /api/npcs/:id` или `PATCH /api/npcs/:id`

Поведение:

- требует авторизацию,
- доступно любому авторизованному пользователю,
- сохраняет весь редактируемый документ,
- работает по модели `last write wins`,
- не использует optimistic versioning,
- возвращает обновлённый `NpcDto`.

### Удаление NPC

- `DELETE /api/npcs/:id`

Поведение:

- требует авторизацию,
- доступно любому авторизованному пользователю,
- soft-delete NPC,
- каскадно удаляет записи из `chronicle_npcs`,
- не трогает уже созданные `combat_states.npcs`.

### NPC, привязанные к хронике

- `GET /api/chronicles/:id/npcs?search=<name>`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- возвращает только NPC, привязанные к этой хронике,
- поиск по имени case-insensitive substring,
- сортировка по имени.

### Кандидаты на привязку к хронике

- `GET /api/chronicles/:id/npcs/available?search=<name>`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- возвращает только NPC, ещё не привязанные к этой хронике,
- поиск по имени,
- сортировка по имени.

### Привязка NPC к хронике

- `POST /api/chronicles/:id/npcs`

Body:

- `{ npcId: string }`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- создаёт запись в `chronicle_npcs`,
- повторная привязка не создаёт дубль.

### Отвязка NPC от хроники

- `DELETE /api/chronicles/:id/npcs/:npcId`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- удаляет только привязку,
- не удаляет сам NPC,
- не удаляет уже существующие NPC-копии из боя.

### Добавление NPC в бой

- `POST /api/chronicles/:id/combat/npcs`

Body:

- `{ npcId: string }`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- разрешено только для NPC, уже привязанных к хронике,
- создаёт snapshot-копию внутри `combat_states`,
- инкрементирует `npcCopyCounters[npcId]`,
- формирует `displayName = "<name> #<ordinal>"`,
- возвращает созданный `CombatNpcDto`.

### Изменение NPC в бою

- `PATCH /api/chronicles/:id/combat/npcs/:combatNpcId`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- обновляет только боевую копию,
- не меняет исходный NPC.

Поддерживаемые поля:

- `initiative`,
- `health`,
- `dead`.

### Удаление NPC-копии из боя

- `DELETE /api/chronicles/:id/combat/npcs/:combatNpcId`

Поведение:

- требует авторизацию,
- доступно только автору хроники,
- удаляет только конкретную копию из текущего боя,
- не затрагивает другие копии того же NPC,
- не меняет `npcCopyCounters`.

### Combat bootstrap

Текущий `GET /api/chronicles/:id/combat` должен начать возвращать:

- `npcs: CombatNpcDto[]` вместо текущего сценария ручных `enemies`,
- данные NPC-копий для отображения всем участникам боя.

Если legacy-поле `enemies` временно останется в документе БД, клиент больше не должен на него опираться.

## Step-by-Step Plan

### Step 1. Зафиксировать отдельный NPC-domain и не пытаться встраивать его в `Character`

1. Зафиксировать, что NPC не наследует текущий wizard/game/storyteller pipeline.
2. Отделить NPC от `Character` на уровне:
- Mongo-модели,
- DTO,
- route layer,
- client pages.
3. Не использовать Socket.IO и `useCharacterSocket` для NPC.
4. Отдельно зафиксировать простую range-validation без V20-derived recalculation.

Done criteria:

- В архитектуре нет попытки запихнуть NPC в `CharacterModel`.
- У NPC есть самостоятельная схема, DTO и CRUD.
- Нет зависимости NPC-редактора от current character socket flow.

### Step 2. Backend: ввести модели `Npc` и `ChronicleNpcLink`

1. Добавить новую Mongoose-модель `Npc`.
2. Добавить новую Mongoose-модель `ChronicleNpcLink`.
3. Добавить индексы:
- `npcs.deleted`,
- `npcs.createdAt`,
- `chronicle_npcs.chronicleId + npcId` unique,
- `chronicle_npcs.chronicleId`,
- `chronicle_npcs.npcId`.
4. Экспортировать новые модели через `apps/server/src/db/index.ts`.
5. Добавить server-side helpers:
- normalizers,
- validators,
- DTO/presenter helpers.

Done criteria:

- В БД есть отдельная коллекция NPC.
- Many-to-many связь с хрониками хранится отдельно и не через `meta.chronicleId`.
- Сервер умеет безопасно искать и удалять привязки.

### Step 3. Backend: реализовать CRUD и поиск по NPC

1. Добавить новый route module, например `apps/server/src/routes/npcs.ts`.
2. Реализовать:
- `GET /npcs`,
- `POST /npcs`,
- `GET /npcs/:id`,
- `PUT/PATCH /npcs/:id`,
- `DELETE /npcs/:id`.
3. Добавить простую валидацию диапазонов для:
- traits,
- resources,
- health track.
4. Поиск по имени сделать на сервере:
- case-insensitive,
- substring,
- trim-safe.
5. При удалении NPC:
- soft-delete NPC,
- удалить все `ChronicleNpcLink`,
- не трогать боевые snapshot-копии.

Done criteria:

- Каталог NPC работает независимо от персонажей игроков.
- Любой авторизованный пользователь может создать, открыть, отредактировать и удалить NPC.
- Поиск по имени работает на сервере.

### Step 4. Backend: привязка NPC к хронике с author-only доступом

1. Добавить helper проверки автора хроники:
- `chronicle.createdByUserId === req.auth.user.id`.
2. Реализовать author-only endpoints:
- `GET /chronicles/:id/npcs`,
- `GET /chronicles/:id/npcs/available`,
- `POST /chronicles/:id/npcs`,
- `DELETE /chronicles/:id/npcs/:npcId`.
3. Гарантировать, что:
- неавтор хроники не видит этот список,
- не может получить список кандидатов,
- не может привязать или отвязать NPC.
4. На сервере отфильтровать только:
- живые NPC,
- не привязанные к хронике для `available`.
5. Сортировку по имени делать на сервере.

Done criteria:

- Привязка NPC к хронике работает как M:N.
- Автор хроники управляет связями, остальные пользователи не видят этот API-слой.
- Повторная привязка не создаёт дубликаты.

### Step 5. Backend: заменить ручных противников в бою на NPC snapshot flow

1. Перестроить `CombatState`:
- заменить текущий target flow `enemies`,
- добавить `npcs`,
- добавить `npcCopyCounters`.
2. Реализовать `POST /chronicles/:id/combat/npcs`:
- проверить авторство хроники,
- проверить наличие привязки `chronicle <-> npc`,
- взять snapshot значений NPC,
- вычислить следующий ordinal,
- создать независимую боевую копию.
3. Реализовать `PATCH /chronicles/:id/combat/npcs/:combatNpcId`:
- инициатива,
- здоровье,
- `dead`.
4. Реализовать `DELETE /chronicles/:id/combat/npcs/:combatNpcId`.
5. Отключить целевое использование старых `/combat/enemies` endpoints и `CombatEnemyDto`.
6. Обновить `GET /chronicles/:id/combat`, чтобы он возвращал NPC-копии всем участникам боя.

Done criteria:

- В бой добавляются только привязанные NPC.
- Каждая копия независима.
- Нумерация копий корректна и не переиспользуется.
- Ручные противники больше не являются целевым механизмом.

### Step 6. Frontend: маршруты и страницы NPC

1. Добавить маршруты:
- `/npcs`,
- `/npcs/new`,
- `/npcs/:id`,
- `/npcs/:id/edit`.
2. Добавить страницу каталога NPC:
- поиск по имени,
- список NPC,
- сортировка по новым,
- кнопка `Создать NPC`.
3. Добавить read-only страницу NPC:
- отображение характеристик,
- кнопка `Редактировать`,
- кнопка удаления с confirm.
4. Добавить отдельную страницу создания/редактирования NPC:
- локальная форма,
- явное `Сохранить`,
- `Отмена/Назад`,
- no autosave.
5. После `POST /npcs` делать redirect на `/npcs/:id`.

Done criteria:

- Есть отдельный глобальный раздел NPC.
- Создание, просмотр и редактирование разделены по страницам.
- Пользовательский flow соответствует зафиксированному UX.

### Step 7. Frontend: NPC-редактор в стиле Storyteller, но на локальной форме

1. Построить новый NPC editor page на основе визуального паттерна `StorytellerPage`, но не переиспользовать её socket semantics.
2. Использовать существующие словари:
- clans,
- sects,
- attributes,
- abilities,

## Risks and Nuances

- Risk: попытка переиспользовать `CharacterModel` приведёт к лишней сложности и регрессиям.
Mitigation: держать NPC отдельной простой сущностью без wizard/socket/derived-логики.

- Risk: current combat schema завязана на `enemies`, а новый flow требует snapshot-копий с нумерацией.
Mitigation: ввести отдельное поле `npcs` и `npcCopyCounters`, а legacy `enemies` перестать использовать в клиенте.

- Risk: старые активные бои могут содержать ручных `enemies`.
Mitigation: либо выполнить одноразовый cleanup/migration, либо жёстко очищать legacy enemy state на `combat/start`.

- Risk: скрытие NPC-блока только в UI недостаточно.
Mitigation: все chronicle NPC routes и NPC combat mutation routes должны дополнительно проверять автора хроники на сервере.

- Risk: global edit/delete для любого авторизованного пользователя повышает шанс случайной порчи NPC.
Mitigation: явные confirm-dialog для удаления, read-only detail как default landing page, а не моментальный inline edit.

- Risk: если хранить у NPC те же `LayeredValue`, что у `Character`, редактор и сервер начнут тянуть ненужные V20-зависимости.
Mitigation: хранить у NPC простые числа и отдельный validator.

- Risk: health validation легко сломать на границах track.
Mitigation: переиспользовать текущую логику health clamping/track semantics там, где это возможно, но без привязки к `Character`.

- Risk: в бою нужна монотонная нумерация копий даже после удаления предыдущих экземпляров.
Mitigation: хранить `npcCopyCounters` в `CombatState`, а не вычислять номер только по текущему массиву участников.

- Risk: feature затронет и client, и server, и shared DTO.
Mitigation: вести работу slices-ами и не начинать combat migration до готового NPC domain и chronicle binding layer.

## Definition of Done (Feature 07)

- В проекте есть отдельная подсистема NPC, не завязанная на `Character`.
- На главной странице есть переход в раздел `Неигровые персонажи`, но нет preview-списка NPC.
- Есть страницы:
  - каталог NPC,
  - создание NPC,
  - read-only просмотр NPC,
  - редактирование NPC.
- Любой авторизованный пользователь может создать, просмотреть, отредактировать и удалить любого NPC.
- NPC хранятся в отдельной коллекции и используют простую ручную модель данных без wizard/socket/V20-derived logic.
- Привязка NPC к хроникам реализована через отдельную many-to-many таблицу.
- На странице хроники author-only NPC block позволяет:
  - видеть привязанных NPC,
  - искать их по имени,
  - привязывать новых NPC через модалку,
  - отвязывать NPC.
- На боевом экране ручные противники заменены на NPC-based flow.
- В бой можно добавить только NPC, уже привязанных к хронике.
- Добавление NPC в бой создаёт независимую snapshot-копию с монотонной нумерацией имени.
- Все пользователи боя видят NPC-копии, но управляет ими только автор хроники.
- Можно удалить конкретную NPC-копию из боя без удаления остальных.
- Удаление исходного NPC каскадно отвязывает его от хроник, но не убирает уже созданные боевые snapshot-копии.
- Обновлены DTO, help-тексты и UI под новый NPC-flow.
- Проходят:
  - `npm run typecheck`,
  - `npm -w apps/server run test`.
