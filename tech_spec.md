# Техническое задание (ТЗ) — **Vampire Siesta** (MVP)

## 0) Кратко о продукте

**Vampire Siesta** — веб-приложение для создания и ведения **листа персонажа Vampire: The Masquerade 20th Anniversary (V20)**.

- **Frontend:** React + Vite + TypeScript, чистый CSS
    
- **Backend:** Node.js + Express + TypeScript + Socket.IO
    
- **DB:** MongoDB + Mongoose
    
- **Доступ:** без пользователей/логина. Любой, у кого есть ссылка на персонажа, может смотреть/редактировать в рамках прав текущего режима.
    
- **Режимы:**
    
    1. **Wizard** — одноразовый помощник создания (строгие правила по книге)
        
    2. **Игра** — “живой” лист (редактируются только ресурсы и тексты)
        
    3. **ST mode** — страница “Storyteller” (можно менять всё, но итоговые значения в допустимых диапазонах)
        

---

## 1) Цели и границы MVP

### Цели

- Создание персонажа V20 через пошаговый wizard с подсказками, бюджетами и строгой валидацией.
    
- Удобное ведение ресурсов в игре (Blood Pool, Willpower, Humanity, Health) с **realtime-синхронизацией** между открытыми вкладками/игроками (Socket.IO).
    
- Возможность “ручного вмешательства” через ST mode без ограничений создания (но с диапазонами).
    
- Группировка персонажей по **хроникам**.
    

### Не входит в MVP

- Пользователи, роли, авторизация, приватность, защита от ботов.
    
- XP-механика и трата опыта (XP полностью отсутствует).
    
- Real-time обновления списков хроник (там только HTTP/F5).
    
- Импорт/экспорт “между разными инсталляциями” без совпадающих справочников/хроник (импорт строгий).

## 2) Основные сущности

### 2.1. Chronicle (Хроника)

- Хранится в Mongo, доступна всем.
    
- На главной показывается список всех хроник.
    
- По умолчанию существует хроника **«Без хроники»** (создаётся seed-скриптом).
    

### 2.2. Character (Персонаж)

- Открывается по **UUID** (неперебираемый ключ в URL).
    
- Имеет флаги `creationFinished` и `deleted`.
    
- В зависимости от `creationFinished`:
    
    - `false` → открывается wizard на сохранённом шаге
        
    - `true` → открывается игровой режим
        

### 2.3. Справочники (в Mongo, отдельными коллекциями)

- `clans`, `disciplines`, `attributes`, `abilities`, `backgrounds`, `virtues`, `merits`, `flaws`, `sects`, `natures`, `demeanors`, `generations`
    
- В персонаже хранятся **строковые `key`**, UI подтягивает `labelRu` из справочников.
    

---

## 3) Страницы и роуты (React Router)

- `/` — Главная:
    
    - список хроник (включая «Без хроники»)
        
    - кнопка **«Создать персонажа»** (создаёт в «Без хроники» и открывает `/c/:uuid`)
        
    - кнопка **«Импорт»** (создаёт нового персонажа, затем импортирует JSON)
        
    - ссылка **Help** (только на главной)
        
- `/help` — инструкция по режимам, импорт/экспорт, общие правила
    
- `/chronicles/:id` — страница хроники:
    
    - список персонажей (без удалённых)
        
    - сортировка по имени персонажа
        
    - для безымянных показываем **«(Без имени)»**
        
    - если `creationFinished=false` — пометка **«Черновик»** + иконка
        
- `/c/:uuid` — персонаж:
    
    - если `creationFinished=false` → Wizard
        
    - иначе → Игровой режим
        
- `/c/:uuid/st` — Storyteller mode
    
- `404` — если персонаж не найден или `deleted=true`

## 4) UX: режимы и доступность действий

### 4.1 Wizard (одноразовый)

- Запускается при `creationFinished=false`.
    
- Сверху — **stepper из подписанных кружков**:
    
    - текущий шаг подсвечен
        
    - кликом можно перейти **только назад** (вперёд — только кнопкой **«Далее»**)
        
- Переход “вперёд”:
    
    - кнопка **«Далее»** запускает валидацию (клиент + сервер)
        
    - если ок — шаг увеличивается
        
    - если ошибки — отображаем тост + подсветка полей (см. формат ошибок ниже)
        
- Если пользователь изменил данные на более раннем шаге:
    
    - сервер принудительно устанавливает `wizard.currentStep = N` (шаг изменения)
        
    - freebie-вложения, сделанные “после”, могут быть откатаны (см. раздел 7)
        

**После нажатия «Завершить»**:

- wizard больше недоступен (вернуться нельзя)
    
- любые правки листа — через ST mode
    

### 4.2 Игровой режим

Редактируемо:

- Blood Pool (dots + число)
    
- Willpower (dots + число)
    
- Humanity (dots + число)
    
- Health (трек 7 уровней; ввод через клики; хранение — как b/l/a числа)
    
- Notes и Equipment (текстовые поля, realtime, last-write-wins)
    

Только чтение:

- все характеристики/дисциплины/фоны/добродетели/мериты/флавы и т.д.
    

Кнопки на странице персонажа (игра):

- **Скопировать ссылку**
    
- **Экспорт JSON**
    
- **Импорт JSON (перезаписать текущего)**
    
- **Перейти в ST mode**
    

### 4.3 Storyteller mode (ST)

- Открывается по `/c/:uuid/st` кнопкой с страницы персонажа.
    
- Доступен **всем, у кого есть ссылка**.
    
- Можно менять **что угодно**, но:
    
    - итоговые значения (`total`) должны оставаться в допустимых диапазонах (см. валидацию)
        
    - изменения статов записываются **только в слой `storyteller`** (base/freebie не меняются)
        
- Если в ST меняется клан — дисциплины, не входящие в новый клан, **сбрасываются**.
    

Также в ST есть:

- кнопка **«Удалить персонажа»** → подтверждение → soft delete (`deleted=true`)
    
    - после удаления `/c/:uuid` возвращает `404`
        
    - восстановление через UI не требуется (вручную в Mongo при необходимости)

## 5) Wizard: шаги и правила создания (V20, зафиксированные для MVP)

### 5.1 Шаги Wizard

1. **Основное**
    
2. **Атрибуты**
    
3. **Способности**
    
4. **Дисциплины**
    
5. **Фоны (Backgrounds)**
    
6. **Добродетели**
    
7. **Достоинства / Недостатки (Merits/Flaws)**
    
8. **Freebies**
    
9. **Финальный обзор + Завершение**
    

Notes/Equipment в wizard **не показываем**.

### 5.2 Обязательные поля “Основное”

- Имя персонажа
    
- Имя игрока
    
- Клан (clanKey)
    
- Поколение (8..14)
    
- Хроника (chronicleId; по умолчанию «Без хроники»)
    
- Секта (sectKey)
    
- Натура (natureKey)
    
- Поведение (demeanorKey)
    

Необязательное:

- Сир (текст)
    
- Концепт (текст)
    
- прочие текстовые поля (можно расширять без ограничений длины, кроме базовых required)
    

### 5.3 Распределение базовых очков (base)

- **Атрибуты:** группы Physical/Social/Mental = **7/5/3** (выбор приоритетов на том же экране)
    
    - Все 9 атрибутов имеют минимум **1**
        
    - Исключение: **Nosferatu → Appearance = 0** (через правило клана)
        
    - Максимум атрибутов: **5**
        
- **Способности:** Talents/Skills/Knowledges = **13/9/5** (выбор приоритетов на том же экране)
    
    - Минимум = 0, максимум = 5
        
    - Ограничение “не больше 3” **не применяется** в MVP
        
- **Дисциплины:** **3** базовые точки
    
    - Только **клановые дисциплины**
        
    - На старте: значение каждой дисциплины **не выше 3**
        
- **Фоны:** **5** базовых точек
    
- **Добродетели:** минимум **1/1/1**, плюс распределяем **7** дополнительных очков
    
    - Conscience / Self-Control / Courage
        
    - Максимум добродетели = 5
        

### 5.4 Производные формулы при завершении

- `startingHumanity = Conscience + Self-Control`
    
- `startingWillpower = Courage`
    

### 5.5 Freebies

- Базово: `15 freebie`
    
- Flaws добавляют freebie, но максимум: `+7` (сверх можно брать flaws, но очков не дают)
    
- На шаге Freebies — **единая страница** со всем листом и возможностью докупить:
    
    - Attributes, Abilities, Disciplines, Backgrounds, Virtues
        
    - Merits (тратят freebie)
        
    - Flaws (дают freebie до лимита +7)
        
- При “перепокупках”/удалении flaws с уже потраченными очками:
    
    - система **автоматически откатывает freebie-вложения** по приоритетному списку (см. раздел 7)
        

### 5.6 Финальный обзор и “сгорание” очков

- “Завершить”:
    
    - если есть ошибки/не заполнены обязательные поля → **запрещено завершать**
        
    - если остались непотраченные очки → показываем предупреждение и требуем подтверждение
        
    - непотраченные очки **сгорают** (в данных не храним “сколько сгорело”)

## 6) Данные: модель “слоёв” и totals

### 6.1 Слои у каждого trait

Для большинства характеристик храним:

type LayeredValue = {  
  base: number;  
  freebie: number;  
  storyteller: number; // может быть отрицательным  
};

**Total не храним в Mongo** — считаем на клиенте:  
`total = base + freebie + storyteller`

В UI точки окрашиваются по источникам (base/freebie/storyteller).

### 6.2 Где слой `storyteller`

В ST mode UI изменяет **итог** (через +/– и клики по dots), а приложение автоматически пересчитывает `storyteller`, чтобы итог совпал:  
`storyteller = desiredTotal - (base + freebie)`

## 7) Откаты freebie (детерминированный rollback)

### 7.1 Когда применяется

- Удалили flaw и бюджет стал отрицательным
    
- Изменили ранние шаги wizard (так, что текущие freebie распределения стали невалидными)
    

### 7.2 Порядок отката freebie (первым откатываем это)

1. Humanity
    
2. Willpower
    
3. Backgrounds
    
4. Disciplines
    
5. Abilities
    
6. Attributes
    
7. Virtues
    
8. Merits (последними)
    

Правило: идём по списку, уменьшаем `freebie`-часть, пока бюджет не станет валидным.

## 8) Resources и Health

### 8.1 Resources (редактируются в игре)

resources: {  
  bloodPool: { current: number };  
  willpower: { current: number };  
  humanity: { current: number };  
  health: { bashing: number; lethal: number; aggravated: number };  
}

Ограничения:

- bloodPool.current: `0..derived.bloodPoolMax`
    
- willpower.current: `0..10`
    
- humanity.current: `0..10`
    
- health: каждое `0..7`, и `bashing + lethal + aggravated <= 7` (строго)
    

### 8.2 UI Health (7 ячеек, последовательные клики)

- Ячейки кликаются **только последовательно** (следующая пустая или последняя заполненная).
    
- Клик **циклически меняет тип** (пусто → bashing → lethal → aggravated → пусто).
    
- В данных хранится не массив ячеек, а **3 числа** b/l/a.
    
- На клиенте строится “нормализованный трек” из чисел (фиксированный порядок отображения).
    
- **Wound penalty** считается на клиенте по стандартной таблице V20 и показывается рядом.

## 9) Derived (храним в документе персонажа)

В `derived` храним:

- `bloodPoolMax` (из поколения)
    
- `bloodPerTurn` (из поколения)
    
- `willpowerMax = 10`
    
- `humanityMax = 10`
    
- `startingHumanity` (фиксируется при finish)
    
- `startingWillpower` (фиксируется при finish)
    

Важно:

- При изменениях поколения в ST пересчитываем `bloodPoolMax/bloodPerTurn`, но **не трогаем** `starting*`.
    

Таблица поколений хранится в коллекции `generations`:

{ generation: number; bloodPoolMax: number; bloodPerTurn: number }

## 10) Валидация

### 10.1 Общие диапазоны (сервер enforce)

- Атрибуты: `1..5` (кроме Appearance, которое может быть `0..5` если клан фиксирует 0)
    
- Способности: `0..5`
    
- Дисциплины: `0..5`, только клановые ключи
    
- Backgrounds: `0..5`
    
- Virtues: `1..5`
    
- Willpower/Humanity: `0..10`
    
- Blood Pool current: `0..derived.bloodPoolMax`
    
- Health sum: `<= 7`
    

### 10.2 Wizard-валидация

- При “Далее” валидируются:
    
    - текущий шаг (полная)
        
    - базовые диапазоны по изменённым полям
        
- При “Завершить” валидируется **весь лист** + бюджеты.
    

### 10.3 Ошибки валидации (формат)

Сервер возвращает массив:

[  
  { "path": "meta.name", "message": "Имя обязательно" },  
  { "path": "traits.disciplines.auspex.base", "message": "..." }  
]

## 11) Realtime: Socket.IO

### 11.1 Комнаты

- Комната на персонажа: `room = characterUuid`
    
- Socket используется только на странице персонажа (wizard/игра/st), не на хронике.
    

### 11.2 Версионирование

- В Character хранится `version: number` (инкремент на каждое изменение)
    

### 11.3 Патчи

Клиент отправляет `set`-патч:

type Patch = {  
  characterUuid: string;  
  baseVersion: number;  
  op: "set";  
  path: string;   // dot-notation  
  value: unknown;  
};

### 11.4 Поведение клиента (Optimistic UI)

- Клиент **сразу применяет** изменение локально.
    
- Отправляет патч на сервер.
    
- Если сервер **reject**:
    
    - показываем тост
        
    - делаем **полный resync**: `GET /api/characters/:uuid`
        
- Если сервер **accept**:
    
    - сервер сохраняет, увеличивает `version`
        
    - сервер broadcast’ит в комнату событие “patchApplied”
        
    - все клиенты (включая отправителя) применяют broadcast как источник истины
        

### 11.5 Серверная обработка патча

Алгоритм:

1. загрузить персонажа из Mongo по uuid
    
2. применить патч к объекту в памяти
    
3. выполнить custom-валидацию (режим + диапазоны + ограничения)
    
4. пересчитать derived при необходимости
    
5. `save()`, увеличить `version`
    
6. broadcast в room

## 12) REST API (Express)

### 12.1 Chronicles

- `GET /api/chronicles` — список хроник (включая «Без хроники»)
    
- `GET /api/chronicles/:id`
    
- `GET /api/chronicles/:id/characters` — список персонажей (без `deleted`), сортировка по имени
    

### 12.2 Characters

- `POST /api/characters` — создать нового персонажа с дефолтами (в «Без хроники», wizard)
    
- `GET /api/characters/:uuid` — получить персонажа (404 если deleted)
    
- `DELETE /api/characters/:uuid` — soft delete (deleted=true)
    

Импорт/экспорт:

- `GET /api/characters/:uuid/export` — вернуть JSON (полный документ **без uuid**)
    
- `POST /api/characters/:uuid/import` — перезаписать персонажа данными из JSON (uuid игнорируется)
    
    - импорт **строгий**: если JSON невалидный → reject
        

### 12.3 Wizard-операции

- `POST /api/characters/:uuid/wizard/next` — валидировать шаг и увеличить `wizard.currentStep`
    
- `POST /api/characters/:uuid/wizard/back` — уменьшить шаг
    
- `POST /api/characters/:uuid/wizard/goto` — перейти на конкретный предыдущий шаг (для клика по stepper), только `targetStep <= currentStep`
    
- `POST /api/characters/:uuid/wizard/finish`
    
    - если есть непотраченные очки → сервер отвечает `warning` и требует `confirmBurn=true` во втором вызове
        
    - при успехе: `creationFinished=true`, фиксирует `starting*`, удаляет `wizard`
        

### 12.4 Dictionaries (по отдельности)

- `GET /api/clans`
    
- `GET /api/disciplines`
    
- `GET /api/attributes`
    
- `GET /api/abilities`
    
- `GET /api/backgrounds`
    
- `GET /api/virtues`
    
- `GET /api/merits`
    
- `GET /api/flaws`
    
- `GET /api/sects`
    
- `GET /api/natures`
    
- `GET /api/demeanors`
    
- `GET /api/generations`
    

Клиент грузит **все** справочники один раз при старте и держит в памяти.

## 13) Дефолты при создании персонажа

При `POST /api/characters` создаётся документ:

- `chronicleId` = «Без хроники»
    
- `creationFinished=false`
    
- `wizard.currentStep=1`
    
- `generation=13` (по умолчанию)
    
- Атрибуты: все `base=1` (кроме Appearance — зависит от клана; пока клан не выбран, `base=1`)
    
- Добродетели: `base=1` для всех трёх
    
- Остальные значения `base=0`
    
- Freebie слой везде `0`
    
- Storyteller слой везде `0`
    
- Resources:
    
    - bloodPool.current=0
        
    - willpower.current=0
        
    - humanity.current=0
        
    - health b/l/a = 0
        
- Notes/Equipment = пусто
    
- derived пересчитан по поколению

## 14) Soft delete

- `deleted=true`, `deletedAt=Date`
    
- Не показываем удалённых в списках хроник.
    
- `GET /api/characters/:uuid` и `/c/:uuid` → **404** для deleted.

## 15) Seed (Mongo наполнение)

В репозитории есть `npm run seed`, который идемпотентно создаёт:

- хронику **«Без хроники»**
    
- базовые справочники V20 (key + labelRu + необходимые поля)
    
- `generations` (8..14) с `bloodPoolMax`, `bloodPerTurn`
    
- `clans` со списком `disciplineKeys` и правилами:
    
    - для Nosferatu: `rules.appearanceFixedTo=0`
        

Админки нет — справочники/хроники можно править вручную в Mongo.
