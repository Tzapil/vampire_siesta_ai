# Фича 03: Popover-описания вместо help-icon и расширение словарных описаний

## Problem Statement

Сейчас справочные подсказки в интерфейсе завязаны на `help-icon` c `title` по hover:
- это неудобно на desktop и особенно плохо работает на touch-устройствах,
- длинные описания трудно читать, потому что они превращаются в неуправляемый browser tooltip,
- часть словарных сущностей вообще не имеет доступной точки входа к описанию в UI,
- у дисциплин отсутствуют полноценные описания с кратким разбором возможностей по уровням,
- часть словарей с описаниями уже лежит в `data`, часть всё ещё хардкодится в `seed`, а часть описаний не доходит до API и клиента.

В итоге пользователь получает непоследовательный UX, а база словарных данных уже достаточно выросла, чтобы требовать более удобного и единообразного способа чтения описаний.

## Goal

Заменить hover-only `help-icon` на click-based popover с читаемым описанием, добавить доступ к описаниям во всех релевантных местах детального UI и расширить словарные данные:
- для дисциплин добавить краткие описания и возможности на 1-5 точках по VtM 20,
- для остальных словарных групп проверить и при необходимости расширить описания,
- обеспечить, чтобы повторный `npm run seed` обновлял описания и новые словарные значения в базе,
- добавить новую секту `Анархи`.

## Non-Goals (для этой фичи)

- Полный энциклопедический перенос правил V20 или дословное цитирование книги.
- Отдельная структурированная модель `levels[]` для дисциплин; в рамках фичи достаточно одной многострочной `description`.
- Подробные описания путей для `Thaumaturgy` и `Necromancy`.
- Превращение `concept` в словарь или popup-выбор; `Концепт` остаётся свободным текстовым полем.
- Автоматическая защита на будущее в виде обязательных тестов/валидаторов полноты `description` для всех словарей.

## Зафиксированные продуктовые решения

- `help-icon` заменяется на кнопку, которая открывает popover по клику.
- Popover открывается под соответствующей строкой или полем и отображается поверх остального контента.
- Popover закрывается по клику в любое место вне него.
- Одновременно открыт только один popover.
- Popover растёт вниз до нижней границы экрана; если высоты не хватает, внутри появляется scroll.
- Контент popover должен сохранять переносы строк и визуально разделять блоки текста, чтобы длинные описания не сливались.
- В popover выводится тот же состав данных, что и в старых tooltip, где он уже был:
- `description`,
- `specializationDescription` и `specializationAt`,
- `category`,
- `pageRef`.
- Замена старого `help-icon` выполняется во всех местах приложения, где он используется сейчас.
- Дополнительно нужно добавить кнопки описания в тех местах, где описания нужны, но сейчас точки входа нет:
- `backgrounds`,
- `virtues`,
- `sects`,
- `natures`,
- `demeanors`.
- Для `Секты`, `Натуры` и `Поведения`, которые выбираются через `select`, кнопка описания ставится рядом с полем и показывает описание текущего выбранного значения.
- Для дисциплин описание хранится одной строкой, но оформляется с переносами строк и явным разделением по уровням 1-5.
- Для `Thaumaturgy` и `Necromancy` делается исключение: только базовое описание дисциплины без описания путей.
- Тексты должны быть авторским кратким пересказом эффектов и условий применения на русском, а не дословным копированием книги.
- В контент-аудит входят все словарные группы, кроме `concepts`, который пока остаётся вне словарного UI:
- `disciplines`,
- `attributes`,
- `abilities`,
- `backgrounds`,
- `virtues`,
- `merits`,
- `flaws`,
- `sects`,
- `natures`,
- `demeanors`.
- В рамках фичи добавляется секта `Анархи`.
- Повторный `npm run seed` должен обновлять уже существующие записи словарей в MongoDB без ручной чистки базы.

## Step-by-Step Plan

### Step 1. Аудит текущего UI и словарного пайплайна

1. Зафиксировать все текущие места использования `help-icon`:
- `Wizard`,
- `GameMode`,
- связанные списки достоинств/недостатков и trait-блоки.
2. Зафиксировать места, где словарь уже показывается пользователю, но у него ещё нет точки входа к описанию:
- `backgrounds`,
- `virtues`,
- `sects`,
- `natures`,
- `demeanors`,
- детальные read-only экраны персонажа и storyteller view.
3. Зафиксировать текущие источники данных:
- `attributes`, `abilities`, `backgrounds`, `disciplines`, `merits`, `flaws`, `archetypes` идут из `data/*.json`,
- `virtues` и `sects` сейчас задаются inline в `apps/server/src/seed.ts`,
- `natures` и `demeanors` берутся из `archetypes.json`, но их `description` сейчас не пробрасывается в словарные API,
- `concepts.json` существует в `data`, но в пользовательском потоке не используется как словарь.

Done criteria:
- Понятен полный список UI-экранов, которые надо затронуть.
- Понятно, где именно теряются описания между `data` -> `seed` -> `API` -> `client`.

### Step 2. Базовый reusable popover-компонент на клиенте

1. Сделать общий компонент или пару `button + popover`, чтобы не дублировать логику по `Wizard`, `GameMode` и другим экранам.
2. Перевести поведение с browser `title` на управляемый React popover:
- открытие по клику,
- закрытие по outside click,
- корректный `z-index`,
- якорение под строкой/полем.
3. Поддержать длинные тексты:
- `white-space: pre-line` или эквивалент,
- ограничение по высоте до нижней границы viewport,
- внутренний scroll только как fallback.
4. Привести кнопку к доступному и единообразному поведению:
- реальный `button`,
- понятный `aria-label`,
- признак открытого состояния.

Done criteria:
- Есть один переиспользуемый popover-паттерн вместо разрозненных `span.help-icon`.
- Многострочные описания читаются без browser tooltip.

### Step 3. Полная замена текущих help-icon

1. Заменить все существующие `help-icon` в `Wizard`.
2. Заменить все существующие `help-icon` в `GameMode`.
3. Удалить устаревшую зависимость от `title`-tooltip там, где теперь используется popover.
4. Сохранить текущую информационную наполненность tooltip-сборки:
- описание,
- специализация,
- категория,
- источник.

Done criteria:
- В приложении больше не осталось hover-only `help-icon`.
- Все старые точки доступа к описанию работают через новый popover.

### Step 4. Добавление кнопок описания в места, где их раньше не было

1. Добавить popover-кнопки для `backgrounds` на шагах создания и редактирования персонажа.
2. Добавить popover-кнопки для `virtues` на шагах создания и редактирования персонажа.
3. Добавить popover-кнопки для `backgrounds` и `virtues` в детальных read-only режимах:
- `GameMode`,
- storyteller view,
- другие экраны детального листа персонажа, где сущность показана отдельной строкой.
4. Для `Секты`, `Натуры` и `Поведения` добавить кнопку рядом с `select`, которая показывает описание текущего выбранного значения.
5. Для read-only представлений `Секты`, `Натуры` и `Поведения` добавить такую же доступную точку входа к описанию рядом с отображаемым значением там, где это уместно в детальном экране персонажа.

Done criteria:
- Пользователь может открыть описание всех целевых словарных сущностей из детального UI, а не только для атрибутов/способностей/дисциплин.

### Step 5. Приведение словарных данных к единому источнику и расширение seed

1. Расширить `data/disciplines.json`, добавив `description` для всех дисциплин.
2. Вынести `virtues` и `sects` из inline-констант в `apps/server/src/seed.ts` в data-driven формат:
- отдельные JSON-файлы в `data/`,
- либо эквивалентный централизованный источник, удобный для контентного редактирования.
3. Добавить descriptions для `virtues` и `sects`.
4. Добавить новую запись секты `Анархи`.
5. Сохранить текущую схему для `natures` и `demeanors` через `archetypes.json`, но обеспечить полноценную доставку их `description` до клиента.
6. Обновить `seed.ts`, чтобы он:
- читал новые поля `description`,
- сидировал новые словари или обновлённые источники,
- корректно обновлял уже существующие записи в MongoDB.

Done criteria:
- Все целевые словари имеют единый понятный путь наполнения через seed.
- Повторный запуск `npm run seed` обновляет тексты и добавляет `Анархов`.

### Step 6. Обновление серверных словарных контрактов

1. Проверить и обновить серверные словарные routes так, чтобы `description` доходил до клиента для всех нужных сущностей:
- `disciplines`,
- `virtues`,
- `sects`,
- `natures`,
- `demeanors`.
2. Убедиться, что существующие DTO на клиенте уже совместимы или при необходимости уточнить типы.
3. Не менять `concept`-контракт: это по-прежнему свободное текстовое поле персонажа.

Done criteria:
- Клиент может получить описания всех целевых сущностей через текущий словарный API.
- Для новых popover не нужны ad-hoc запросы или дополнительные endpoint’ы.

### Step 7. Контентный проход по всем словарным группам

1. Подготовить краткие, читаемые, авторские описания для всех дисциплин.
2. Для обычных дисциплин оформить `description` с блоками по уровням 1-5.
3. Для `Thaumaturgy` и `Necromancy` ограничиться общим описанием дисциплины без путей.
4. Провести ревизию и при необходимости расширить существующие descriptions для:
- `attributes`,
- `abilities`,
- `backgrounds`,
- `virtues`,
- `merits`,
- `flaws`,
- `sects`,
- `natures`,
- `demeanors`.
5. Не трогать `concepts.json` и не встраивать его в UI этой фичи.

Done criteria:
- Все целевые словари имеют осмысленные и достаточно полные описания для popover-режима.
- Дисциплины читаются как краткая справка, а не как сплошной текст.

### Step 8. Визуальная и функциональная проверка

1. Проверить popover-поведение на desktop:
- открытие под нужной строкой,
- перекрытие контента,
- закрытие по клику вне popover,
- корректная работа нескольких разных кнопок подряд.
2. Проверить mobile/touch-сценарии:
- открытие по тапу,
- отсутствие зависимости от hover,
- удобочитаемость длинных описаний,
- scroll внутри popover только если реально не хватает высоты.
3. Проверить, что переносы строк из словарных descriptions корректно отображаются в UI.
4. Проверить, что после `npm run seed` новые описания и секта `Анархи` реально попадают в интерфейс.

Done criteria:
- Новый UX работает одинаково предсказуемо на desktop и mobile.
- Многострочные описания не ломают вёрстку и не обрезаются неожиданно.

### Step 9. Техническая верификация

1. Прогнать `npm run seed`.
2. Прогнать `npm run typecheck`.
3. Прогнать `npm -w apps/server run test`.
4. При необходимости добавить или обновить минимальные серверные тесты для словарных routes/seed-обновлений, если текущий набор не ловит регрессии по описаниям.

Done criteria:
- Фича проходит базовую техническую проверку.
- Изменения словарных контрактов и seed не ломают текущий проект.

## Suggested Delivery Slices

1. Slice A: Steps 1-3.
Описание:
Собрать audit, сделать reusable popover и заменить существующие `help-icon` без расширения контента.

2. Slice B: Steps 4-6.
Описание:
Довести UI-покрытие до всех нужных словарных сущностей и привести `seed`/API к единому формату описаний.

3. Slice C: Steps 7-9.
Описание:
Наполнить и вычитать контент, добавить `Анархов`, прогнать seed, typecheck, тесты и финальный smoke-test.

## Risks and Mitigations

- Risk: длинные дисциплинарные описания начнут ломать layout или будут неудобны на маленьком экране.
Mitigation: ограничение popover по высоте viewport, `pre-line`, scroll только как fallback, ручной mobile smoke-test.

- Risk: описания для `virtues`, `sects`, `natures`, `demeanors` останутся в базе, но не дойдут до клиента из-за неполного `.select(...)` в routes.
Mitigation: отдельный шаг на ревизию словарных endpoint’ов и smoke-проверку конкретных полей в UI.

- Risk: `seed` будет пополнять новые описания только для новых записей, но не обновит существующие документы.
Mitigation: сохранить upsert/update-подход и отдельно проверить rerun `npm run seed` на уже заполненной базе.

- Risk: контент фичи разрастётся из-за дисциплин и затянет delivery.
Mitigation: сначала закончить UI и data pipeline, затем делать отдельный целевой контентный проход по словарям.

## Definition of Done (Feature 03)

- Hover-only `help-icon` больше не используется; описания открываются по клику через popover.
- Popover открывается под строкой/полем, показывается поверх контента и закрывается по клику вне него.
- Длинные descriptions корректно читаются с переносами строк и не разваливают вёрстку.
- Для `backgrounds`, `virtues`, `sects`, `natures`, `demeanors` в детальном UI есть доступный способ открыть описание.
- Все дисциплины имеют краткое описание; для обычных дисциплин оно структурировано по уровням 1-5.
- Для `Thaumaturgy` и `Necromancy` добавлено только общее описание дисциплины.
- Описания словарей обновлены и расширены для всех целевых групп, кроме `concepts`.
- В словаре сект присутствует `Анархи`.
- Повторный `npm run seed` обновляет данные в MongoDB и доставляет новые описания в UI.
